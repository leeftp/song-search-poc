# 生成AIへの指示設計ガイド

生成AI（LLM）をシステムに組み込む際の、プロンプト設計のベストプラクティス。  
うたレコ（llm-chat）の実装をもとに整理。

---

## LLMへの入力は4層で構成される

```
┌─────────────────────────────────────────┐
│  1. システムプロンプト                   │
│     「役割・ルール・出力形式」           │
├─────────────────────────────────────────┤
│  2. メモリ・外部知識（RAG）             │
│     「LLMが知らない情報を注入」         │
├─────────────────────────────────────────┤
│  3. コンテキスト（会話履歴）            │
│     user: …  assistant: …  user: …     │
├─────────────────────────────────────────┤
│  4. ユーザーインプット（今回の指示）     │
└─────────────────────────────────────────┘
              ↓
            LLM
              ↓
            出力
```

---

## 各層の役割と設計方針

### 1. システムプロンプト

**「このAIは何者か、何をするか」の定義。会話全体を通して効く土台。**

```
あなたは○○の専門家です。
以下のルールに従って回答してください：
- ルール1
- ルール2
出力形式: JSON で返すこと
```

| 設計ポイント | 内容 |
|---|---|
| 役割を明確に | 曖昧な人格定義はブレの原因になる |
| 制約を書く | 「やってはいけないこと」も明示する |
| 出力形式を指定 | JSON・箇条書きなど、後続処理に合わせて固定する |
| 変わらない部分だけ書く | セッションや状況に依存する情報はここに入れない |

**このプロジェクトの実装:** `clarification.py` の `SYSTEM_PROMPT_TEMPLATE`

---

### 2. メモリ・外部知識（RAG）

**LLMの学習データにない情報を、テキストとしてプロンプトに差し込む仕組み。**

```
[このユーザーの好み]
よく歌う歌手: 米津玄師、back number
好むジャンル: J-Pop

[現在の状況]
気分: 盛り上がる、人数: 5人以上、年代: 20代
```

| 設計ポイント | 内容 |
|---|---|
| 全部入れない | 今回の返答に関係ある情報だけ選んで注入する |
| テキスト化する | DB の生データではなく、LLMが読める文章に変換してから渡す |
| 鮮度を意識する | 古い情報が残るとノイズになる |
| システムプロンプトと分けて管理 | 動的に変わる情報を静的な役割定義と混在させない |

**このプロジェクトの実装:**
- `profile_hint()` … 歌唱履歴・推薦履歴から好みを集計してテキスト化
- `context_hint` … 画面上部のフィルター（気分/感情/地域/人数/年代）の選択値

---

### 3. コンテキスト（会話履歴）

**「ここまでどんな話をしてきたか」をLLMに伝える。**

```
user:      失恋した、泣ける曲がほしい
assistant: 年代の好みはありますか？
user:      90年代で
```

> **重要:** LLM自体に記憶機能はない。渡したものしか知らない。  
> 会話が成立しているように見えるのは、毎回履歴を丸ごと渡しているから。

| 設計ポイント | 内容 |
|---|---|
| 長くなったら切る | トークン上限・コスト対策に直近N件だけ渡す |
| roleを正確に | `user` / `assistant` の区別が崩れると精度が落ちる |
| 要約して圧縮する | 長い履歴はLLMに要約させて1メッセージに圧縮する手もある |

**このプロジェクトの実装:** `session.messages`（in-memory、セッション単位）

---

### 4. ユーザーインプット

**「今、何を頼むか」。ユーザーが毎回送るテキスト。**

| 設計ポイント | 内容 |
|---|---|
| 具体的・短く | 一度に複数の指示を詰め込まない |
| システムプロンプトで解釈を制御 | インプットが曖昧でも、システムプロンプトで動作を定義しておくと安定する |
| 前処理を挟む | 音声認識テキストの整形、フィルター値との結合など |

---

## 「プロンプト」という言葉について

| 呼び方 | 指す範囲 |
|---|---|
| **システムプロンプト** | 役割・ルール・出力形式の土台（変わらない部分） |
| **プロンプト**（広義） | LLMに渡す入力の全体 = 上記4層をまとめたもの |

「プロンプトエンジニアリング」とは、この4層をどう設計・組み合わせるかの技術。

---

## このプロジェクトでの対応関係

| 設計層 | 実装箇所 | 内容 |
|---|---|---|
| システムプロンプト | `clarification.py` `SYSTEM_PROMPT_TEMPLATE` | カラオケAIの役割・判定ルール・JSON出力形式 |
| メモリ | `user_profile.py` `profile_hint()` | 歌唱履歴・推薦履歴から生成したユーザー好みの要約 |
| 外部知識 | `main.py` `context_hint` | 画面フィルター（気分/感情/地域/人数/年代）の選択値 |
| コンテキスト | `session_store.py` `session.messages` | セッション内の会話履歴（in-memory） |
| ユーザーインプット | `POST /chat` `req.message` | チャットに入力されたテキスト |

---

## LLMを使ったシステム設計の注意点

- **LLMはステートレス**: 状態の保持はアプリ側の責務（DB・session）
- **出力は非決定的**: 同じ入力でも毎回同じ出力にはならない。後続処理で検証・フィルタリングを行う
  - このプロジェクトでは `matcher.py` でDB照合し、幻覚（存在しない曲）を除外
- **コストはトークン数に比例**: 4層の合計トークン数を意識して設計する
- **プロンプトはコード**: バージョン管理・テストの対象として扱う
